# フロントエンドのアーキテクチャに関しての所感

## なぜフロントエンドにアーキテクチャを実装するのか

今回のwebアプリのパターンでは以下のようなイベントによって発火する処理が大半。

- ユーザがボタンを押すとスポットの順番が入れ替わり、移動時間や表示が再計算される。
- ユーザがボタンを押すと入力フォームにある場所を検索し、表示する。

特に入力フォームが用いられているイベントでは、空白や文字数の制限などのバリデーションが必要となる。
またアニメーションなどの処理も記述する必要がある。
通常このようなイベントでは、ひとつの関数に処理を記述し`onclick`で処理を呼び出す場合が多いと思うが、今回のようなケースだとひとつの関数に複数の処理を書く必要がある。
もちろん、ひとつの関数でひとつのユースケースを満たすのだから、そのような構造でも問題はないと思う。
しかしデータの取り扱いとアニメーション、バリデーションなどはしっかりと分けて記述した方が保守性が高まる。
特に、サーバサイドから渡される値が変更になった場合、しっかりとしたアーキテクチャを持っていないと潜在的なバグを生み出す原因にもなりかねない。
またフロントサイドはサーバサイドに比べて変更頻度が高く、コードの変更(修正、追加、削除)も多くなる。
そのような場合に、むやみにコードを変更するとこれもまた潜在的なバグを生み出す原因になる。
このような事態を防ぐためにも、コードや仕様の変更に強く、可読性や保守性の高いアーキテクチャをフロントサイドに採用する必要があると考える。
今回はtypescriptによるフロントサイドの実装なので、クリーンアーキテクチャを参考としたアーキテクチャを実装した。
